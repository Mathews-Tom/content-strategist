# Tone Adaptation Guide

Rules for matching voice and style to audience and channel without losing identity.

---

## Audience Profiles

### Developer / Technical Practitioner

| Dimension | Setting |
|-----------|---------|
| Vocabulary | Technical terms without definition, jargon accepted |
| Evidence type | Code examples, benchmarks, architecture patterns |
| Depth | Deep — explain the why behind the how |
| Tone | Peer-to-peer, direct, no marketing language |
| Avoid | Buzzwords, vague ROI claims, executive framing |

### Executive / Decision Maker

| Dimension | Setting |
|-----------|---------|
| Vocabulary | Business outcomes, risk, ROI, timeline, cost |
| Evidence type | Dollar figures, percentages, competitive positioning |
| Depth | Surface with optional deep-dive appendix |
| Tone | Authoritative, concise, results-oriented |
| Avoid | Implementation details, code, architecture jargon |

### General Technical Audience

| Dimension | Setting |
|-----------|---------|
| Vocabulary | Technical concepts with brief context, no deep jargon |
| Evidence type | Analogies, case studies, comparative examples |
| Depth | Medium — explain concepts, skip implementation |
| Tone | Conversational but informed, accessible authority |
| Avoid | Condescension, over-simplification, assumptions of expertise |

## Channel Tone Matrix

| Channel | Register | Sentence Length | Paragraph Length | Personal Pronouns |
|---------|----------|----------------|-----------------|-------------------|
| LinkedIn | Conversational | Short (8-15 words) | 1-2 sentences | "I", "we", "you" freely |
| Blog | Authoritative | Medium (12-20 words) | 2-4 sentences | "you" for reader, "we" for team |
| Slides | Concise | Fragment or short (3-10 words) | N/A (bullets) | Minimal |
| PDF Report | Formal | Medium-long (15-25 words) | 3-5 sentences | "we" for organization, avoid "I" |

## Tone Calibration Patterns

### Making Technical Content Accessible

- Replace jargon with outcome: "event-driven architecture" → "a system that reacts to changes instantly"
- Add a concrete example after every abstract statement
- Use "this means..." bridges between technical fact and practical impact

### Making Business Content Specific

- Replace adjectives with numbers: "significant" → "37%"
- Replace passive with active: "costs were reduced" → "the migration cut costs by $240K/year"
- Replace future-tense promises with evidence: "will improve" → "improved by X in [case study]"

### Making Content Conversational

- Start sentences with "Here's the thing" / "The reality is" / "What we found"
- Use contractions: "it's", "we're", "that's"
- Ask rhetorical questions to maintain engagement
- Break the fourth wall: address the reader directly

### Making Content Formal

- Remove contractions
- Use passive voice sparingly but acceptably for objectivity
- Replace conversational bridges with section headers
- Cite sources with footnotes or inline references

## Anti-Patterns (Remove on Sight)

| Pattern | Problem | Fix |
|---------|---------|-----|
| "In today's rapidly evolving landscape" | Generic AI opener | State the specific context |
| "It's important to note that" | Filler | Delete, state the fact directly |
| "Significant improvement" | Vague | Replace with specific number |
| "Leverage/utilize" | Corporate filler | Use "use" |
| "Game-changing/revolutionary" | Hyperbole | Describe the actual impact |
| "In conclusion" | Formulaic transition | Summarize without announcing it |
| "As we can see" | Padding | Delete |
| "Moving forward" | Vague temporal | State the specific next step |
| Three adjectives in a row | Over-modification | Pick the strongest one |
| Semicolons in casual content | Register mismatch | Use period and new sentence |
